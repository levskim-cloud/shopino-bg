# SHOPINO.BG — онлайн магазин + бекофис платформа

Пълен пакет: витрина за клиенти + бекофис (отделна среда) + Node.js бекенд с база данни във файл.
Без външни библиотеки — трябва само Node.js.

## Какво има вътре
```
server.js           ← бекендът (стартираш само него)
package.json
public/index.html   ← целият фронтенд (витрина + бекофис)
data/store.json     ← базата данни (създава се сама при първото пускане)
```

## Стартиране на твоя компютър (за тест)
1. Инсталирай Node.js LTS от https://nodejs.org (ако го нямаш).
2. Разархивирай папката, отвори терминал в нея и напиши:
   ```
   node server.js
   ```
3. Магазинът: **http://localhost:3000**
   Бекофисът: **http://localhost:3000/#admin** — парола **admin123** (смени я веднага от Настройки → Сигурност).

## Качване на хостинг (да го видиш „на живо“)
Най-лесните варианти за Node.js приложение:

- **Render.com** (има безплатен план): New → Web Service → качваш папката в GitHub или като ZIP → Build command: празно, Start command: `node server.js`. Получаваш адрес от типа shopino.onrender.com с HTTPS.
- **Railway.app** — аналогично, също много лесно.
- **Български хостинг/VPS** (SuperHosting, ICN.bg и др. с Node.js поддръжка): качваш файловете, пускаш `node server.js` (или през pm2, за да върви постоянно: `npm i -g pm2 && pm2 start server.js`).

После закачаш домейна си (напр. shopino.bg) от настройките на хостинга. HTTPS е задължителен за реален магазин — Render/Railway го дават автоматично, при VPS се пуска Let's Encrypt.

⚠ Важно: файлът **data/store.json** е цялата ти база (продукти, поръчки, фактури, клиенти). Прави му редовно копие. На Render безплатният план губи диска при рестарт — за реална работа ползвай план с постоянен диск или VPS.

## Бекофисът управлява всичко без код
Табло с приходи и графика · Поръчки с филтри по статус, търсене, CSV експорт · Продукти (качване със снимка, наличности, „% Отстъпка“ със задраскана цена) · Категории (създаване, скриване, разместване — хедър и фуутър се обновяват сами) · Промо кодове (процент/сума, срок, лимит) · Клиенти (регистрирани и гости) · Маркетинг кампании (съобщения в профила на клиентите; имейлът чака SMTP връзка) · Банери на началната страница · Дизайн (цветове с автоматични нюанси, 3 шрифтови двойки, лого: знак / качен файл / само текст, име на магазина) · Текстове (topbar, заглавия, фуутър) · Страници (Общи условия, Поверителност + нови) · SEO (title/description/keywords с преглед като в Google; бекендът ги вгражда в страницата и дава /sitemap.xml и /robots.txt) · Фактури по чл. 114 ЗДДС с печат/PDF · Настройки (фирма, доставка, цени, профилактика, пароли).

## Интеграция с Еконт
В **Настройки → Интеграция с Еконт** има превключвател ДЕМО/РЕАЛЕН режим.
- В ДЕМО режим номерата на товарителници се генерират локално — за тестване.
- Когато получиш акаунт от **ee.econt.com**: впиши потребител/парола и данните на подателя, изключи демо режима — бутонът „Създай товарителница“ в поръчката ще праща реална заявка `Shipments/LabelService.createLabel.json` към Еконт (с наложен платеж в евро при НП) и ще връща номер + PDF етикет.
- Може да тестваш реални заявки и срещу тестовата им среда demo.econt.com с потребител `iasp-dev` / `1Asp-dev` (вече е попълнен по подразбиране).
- Следваща стъпка от разработчик (по желание): падащо меню с офисите от `Nomenclatures.getOffices` вместо свободен текст.

## Какво още трябва за 100% реална търговия
- **Плащане с карта**: формата е готова визуално; истинското плащане изисква договор с банка/доставчик (Borica, myPOS, Stripe, SumUp) и свързване на техните ключове в бекенда от разработчик (~1–2 дни работа).
- **Имейли** (потвърждения, кампании): бекендът има готова точка за SMTP услуга (Brevo, Mailgun) — вписва се ключ от услугата.
- **Спиди**: същият модел като Еконт — трябва акаунт от api.speedy.bg.
- За по-сериозен трафик: преминаване от store.json към истинска база (SQLite/Postgres) — структурата на API-то остава същата.

Това е работеща основа за тестване и първи продажби, не крайна корпоративна система — прегледай я с разработчик преди да приемаш реални плащания.
